# Clean Code Principles

프로젝트 코드 작성 시 기본 원칙입니다.

## 1. 단일 책임
- 하나의 클래스/함수는 한 가지 이유로만 변경되도록 설계한다.
- 도메인 로직, 인프라 로직, 표현 로직을 분리한다.

## 2. 의미 있는 이름
- 축약어보다 의도를 드러내는 이름을 사용한다.
- Boolean은 `is/has/can`으로 시작한다.

## 3. 작은 함수
- 함수는 짧고 한 가지 일을 하게 유지한다.
- 조건 분기/중복 로직은 private 함수로 분리한다.

## 4. 명시적 계약
- 입력/출력/에러 코드를 문서와 코드에서 일치시킨다.
- 모호한 null 대신 명확한 기본값/예외를 선택한다.

## 5. 불변성 우선
- 가능하면 immutable 객체를 사용한다.
- 상태 변경이 필요하면 변경 지점을 최소화한다.

## 6. 예외 처리 일관성
- 예외는 비즈니스 의미가 드러나는 에러 코드로 변환한다.
- 재시도 가능 여부(`retryable`) 등 운영에 필요한 정보를 응답에 포함한다.

## 7. 테스트 가능성
- 외부 의존성(API/DB/시간/랜덤)은 주입 가능하게 만든다.
- 핵심 유스케이스는 단위 테스트와 통합 테스트로 보호한다.

## 8. 문서 동기화
- API/스키마 변경 시 `DOCS/API_SPECIFICATION.md`를 동시에 수정한다.
- 제거 예정 API는 deprecate 기간을 두고 정리한다.
